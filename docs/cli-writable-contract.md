# PromptHub `ph` CLI — Writable Command Contract (v1)

This document is the source-of-truth decision for what `ph` is allowed to
**write** in v1. It supersedes any ad-hoc behavior in individual writable
commands; downstream implementation tasks (CLI-14 and later) MUST link
back to this contract instead of re-deciding behavior locally.

The read/output contract (JSON shapes, exit codes, identifier precedence,
stdout/stderr policy) lives in [`docs/cli-contract.md`](cli-contract.md).
The app/CLI parity contract for the on-disk format lives in
[`docs/cli-parity.md`](cli-parity.md). This document only covers the
write path.

## 1. Source of truth

| Domain                 | Canonical store                          | CLI may read | CLI may write (v1) |
|------------------------|------------------------------------------|--------------|--------------------|
| Prompts                | PromptHub.app SwiftData (`Prompt` model) | yes          | **yes**            |
| Skill drafts (authoring)| PromptHub.app SwiftData (`Skill` model + `SkillDraftPackageStore`) | yes | **no — deferred to v2** |
| Installed skill copies (per-agent files in `~/.codex/skills`, etc.) | The agent directories themselves | yes | yes (already shipped: install / uninstall / update / reinstall / where) |
| Doctor diagnostics     | derived; no store                        | yes (compute) | n/a              |

The interchange surface between the app and the CLI is the
`~/.prompthub/` directory tree exported by
[`prompthub/Managers/PromptHubBridge.swift`](../prompthub/Managers/PromptHubBridge.swift).
**Both sides are allowed to read and write it.** SwiftData remains the
canonical model store; `~/.prompthub/` is the contract that lets the two
processes share state without IPC or a shared database.

## 2. v1 writable command matrix

| Command                                              | Status      | Touches                                              |
|------------------------------------------------------|-------------|------------------------------------------------------|
| `ph prompt create --name … [--description …] [--body @file.md | --body-stdin] [--id <uuid>]` | **stable v1** | writes `~/.prompthub/prompts/<uuid>.md` |
| `ph prompt update <id|slug> [--name …] [--description …] [--body @file.md | --body-stdin]` | **stable v1** | rewrites `~/.prompthub/prompts/<uuid>.md` |
| `ph prompt delete <id|slug> [--yes]`                | **stable v1** | removes `~/.prompthub/prompts/<uuid>.md`            |
| `ph skill create`                                    | shipped in v1      | exported-package writer for `~/.prompthub/skills/<uuid>/SKILL.md`; intentionally separate from in-app draft editing |
| `ph skill update / delete`                           | deferred to v2     | in-app skill drafts still span multiple rows + lock files; destructive draft mutation stays app-owned for now |
| `ph skill install / uninstall / update / reinstall` | already shipped | per-agent install directories (not `~/.prompthub/`) |
| `ph sync`                                            | **deferred** | the app drives sync on launch; CLI is intentionally not given a way to drive the app |

`stable v1` means the command shape, JSON output, and exit-code behavior
are frozen against schema version `"1"` defined in
`docs/cli-contract.md`.

## 3. Identifier strategy

- **`id`** (prompt and skill): a UUIDv4 string. Stable for the lifetime
  of the asset. Generated by `ph prompt create` when not supplied;
  `--id <uuid>` is allowed for deterministic test fixtures and import
  workflows, but the CLI MUST reject ids that already exist on disk
  (exit non-zero, category `not found` rename → re-use `invalid input`).
- **`slug`**: derived from `name` via the same rule as
  `PromptHubBridge.slug(for:)`. Regenerated on every `update` that
  changes `name`. Never user-editable in v1.
- **`installationName`** (skills only): equal to `slug` today. Not
  writable in v1 because skills are not writable in v1.
- **Identifier precedence on `update` / `delete`**: identical to the
  read precedence documented in `docs/cli-contract.md` §4:
  exact match on `id` / `slug` / `installationName` / `name`, then
  prefix match. Ambiguous matches MUST exit non-zero with category
  `ambiguous match` before any file is modified.

## 4. On-disk format and atomicity

CLI-written prompt files MUST be byte-equivalent in structure to what
`PromptHubBridge.promptMarkdown` produces today (see
[`docs/cli-parity.md`](cli-parity.md) §Exported prompt parity):

```
---
id: <uuidv4>
name: <yaml-scalar>
slug: <derived-from-name>
description: <yaml-scalar>      # optional
link: <yaml-scalar>             # optional, only when carried over from existing file
exported_at: <iso8601>          # always rewritten on write
---

<body>
```

All writes MUST be atomic (`String.write(to:atomically: true, encoding:)`
or equivalent). A failed write MUST NOT leave a partially-written file
in place; the previous version stays on disk if the write fails.

`exported_at` is rewritten on every CLI write to the current time. This
matches the bridge's behavior and makes "last-writer-wins" mtime-based
conflict resolution work both ways.

The optional `link` field is preserved verbatim on `update` even
though the CLI does not expose a flag for it in v1 (see §6).

## 5. Sync timing and conflict semantics

### CLI → app

CLI writes are durable on disk immediately. The app picks them up at:

1. The next `PromptHubBridge.syncAll(prompts:skills:)` run — currently
   invoked on app launch and on explicit "Reload from disk".
2. Whenever a planned **bridge ingest pass** is added (see §7); until
   then, CLI-created prompts are visible to the CLI immediately but
   visible to the app only after the user re-launches or triggers a
   manual reconciliation flow that the app team adds.

The CLI does NOT drive the app. There is no `ph sync` in v1. Users
who want the app to see a CLI write before the next launch must
manually reload from disk in the PromptHub Dashboard.

### App → CLI

Unchanged: the app's `syncAll` writes `~/.prompthub/prompts/<uuid>.md`
and `~/.prompthub/skills/<uuid>/`. The CLI reads them.

### Conflict resolution

- **Last writer wins on a per-file basis.** File mtime is the
  tiebreaker.
- **Prune safety**: `PromptHubBridge.syncAll` prunes prompts/skills
  whose UUID is not in the live SwiftData set. This is a known
  data-loss risk for CLI-created prompts that the app has not yet
  ingested. Until the bridge ingest pass ships (§7), the app MUST NOT
  call `pruneOrphans` while CLI-created prompts are pending. The
  practical mitigation in v1 is documentation: users who create
  prompts via the CLI should re-launch the app before the next
  `syncAll`. CLI-14 surfaces this in `ph prompt create` stderr as a
  one-line hint.

## 6. Failure modes and exit-code mapping

| Failure                                | Category (docs/cli-contract.md §3) | Notes |
|----------------------------------------|------------------------------------|-------|
| `id` collision on `create`             | `invalid input`                    | suggest `ph prompt update` |
| `<id|slug>` not found on `update`/`delete` | `not found`                    | reuses `assetNotFound` from the read path |
| ambiguous identifier on `update`/`delete` | `ambiguous match`               | candidates on stderr; no write performed |
| disk write fails (permissions, full)   | `environmental failure`            | original file untouched |
| `--body @file.md` references missing file | `invalid input`                | error includes the path |
| `--name` rendered slug collides with another live prompt | `invalid input` | suggest distinct name |
| `delete` without `--yes` on TTY        | `safety refusal`                   | matches `unmanagedSkill` pattern from skill uninstall |

`ph prompt delete` MUST refuse to operate without `--yes` when stdout
is a TTY. In non-interactive use (pipes, CI), `--yes` is implied so
scripts do not need to fake a TTY.

## 7. Out of scope for v1 (explicit deferrals)

These are intentionally NOT shipped in v1. They MUST NOT leak into
CLI-14 or any contemporary task without a follow-up contract bump.

1. **`ph skill update/delete` for app-owned drafts.** `ph skill create`
   only writes exported packages under `~/.prompthub/skills/`. Draft
   mutation inside the app still spans multiple SwiftData rows plus a
   `SkillDraftPackageStore` package directory, so destructive edits stay
   deferred to v2.
2. **CloudKit conflict resolution from the CLI.** The app already
   reconciles CloudKit, and the CLI has no signed-in user context.
3. **Live in-app refresh** when the CLI writes. The app reloads on
   `syncAll`; v1 does not push events to a running app.
4. **`ph sync` that drives the app.** Adding a one-way "tell the app to
   reload" command requires an IPC channel we have not designed.
5. **Bidirectional `link` field editing.** The bridge writes `link`
   when present; v1 preserves it on `update` but does not surface a
   `--link` flag.
6. **Multi-process file locking on `~/.prompthub/`.** v1 assumes a
   single user and infrequent writes; concurrent writers race and the
   last writer wins.
7. **Bridge ingest pass** (`PromptHubBridge.importPrompts(from:)` or
   equivalent that promotes CLI-written files into SwiftData rows).
   This is the missing piece that closes the loop. Tracked as a future
   task; CLI-14 ships without it and documents the gap to users.

## 8. Required regression tests for CLI-14

The implementation task for `ph prompt create/update/delete` is
required to add **all** of the following, in addition to the CLI's
existing parity and snapshot suites:

1. **Round-trip read/write** — `ph prompt create --name "Foo" --body @x.md` followed by `ph prompt show foo` returns the same content and ID.
2. **Update only the requested fields** — `ph prompt update foo --description "new"` preserves `body`, `id`, `link`, and regenerates `slug` only when `--name` changes.
3. **Slug regeneration on rename** — `ph prompt update foo --name "Bar"` produces `slug: bar` and leaves `id` unchanged.
4. **Atomic write under failure** — simulate a write failure and assert the previous file is intact and `ph prompt show foo` still returns the old version.
5. **Delete refuses without `--yes` on TTY but accepts in scripts** — assert both behaviors with a controlled stdout stream.
6. **Cross-format parity** — file written by `ph prompt create` is parseable by `PromptHubBridge` reading the same path back (mirror `CLIParityTests.bridgePromptExportCarriesCLIContractFields()`).
7. **Identifier precedence on `update`/`delete`** — exact `id`, exact `slug`, exact `name`, prefix on `id`/`slug`; ambiguous prefix exits with `ambiguous match` and no write happens.
8. **Stderr hint about app re-sync** — `ph prompt create` emits exactly one stderr line pointing at "the running PromptHub app will pick this up on next launch", to make the §5 deferral visible to users.
9. **`--id <uuid>` collision** — supplying an id that already exists on disk fails with `invalid input` and does not overwrite.
10. **`--body @file.md` missing file** — fails with `invalid input` and surfaces the missing path on stderr.

These tests live in
`PromptHubCLI/Tests/PromptHubCLITests/PromptWriteTests.swift` (to be
created by CLI-14) and a smoke companion at
`PromptHubCLI/Tests/Smoke/prompt-write.sh`.

## 9. Versioning policy reminder

Writable command shapes (`ph prompt create/update/delete`) are part of
the v1 contract. Renaming or removing a flag, changing default
behavior (e.g. making `--yes` no longer required), or changing on-disk
format requires a schema-major bump in
`PromptHubCLISchemaVersion`. Adding new flags is allowed within v1.
